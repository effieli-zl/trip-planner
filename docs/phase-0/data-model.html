Trip Planner — Phase 0 Data Model

Normalized, relational. The itinerary is the single source of truth; one uniform Entry record, with people and money split into their own tables so costs and companions can be tracked per person.

Standalone, file-backed (trips/<slug>/itinerary.json), validated against a versioned schema. A durable contract a future app/connector can consume.

schemaVersion 0.2.0 · draft for review

Hierarchy (containment)

Parent → child ownership. Junction tables (below) wire the many-to-many links across branches.

Itinerary — root · schemaVersion
Trip — id, name, start/end date (rollup from sub-tables)
Location — start/end date · overlap warning when 2 locations overlap ≥2 days (dismissible)
Entry — type: Lodging · Transportation · Activity · Dining; subtype, start/end date-time
Finance — total, estimate⇄confirmed, paid, unpaid (derived)
Member — role: Owner · Coordinator · Participant · Follower; traveling flag

Entity-relationship diagram

Crow's-foot notation. | one · < many. Dashed = optional reference. Junctions (Location–Member, Finance–Member) resolve the many-to-many links.

Why the split: Member, LocationMember, and FinanceMember let one entry's cost be divided across people (who's on which leg, who owes what, who paid) without duplicating entries — the relational backbone for group trips.

Relationship (junction) tables

JunctionLinksCard.Extra fieldsPurpose
Trip–MemberTrip · Member* : *role, travelingWho is on the trip and in what capacity. (Held on Member in Phase 0, since members are trip-scoped.)
Location–MemberLocation · Member* : *arrival?, departure?Which members are at which leg — supports converging trips (people arrive/leave a location on different days).
Finance–MemberFinance · Member* : *shareAmount / sharePct, isPayee, settledBill allocation — split a cost per person, mark who fronted it (payee), track settlement.
Bill allocation logic (Finance–Member): each Finance row can be divided among members by explicit amount or percentage (helper: equal split across traveling members). One or more members are flagged isPayee (they paid). "Who owes whom" = each member's shareAmount minus what they paid, netted across the trip. settled marks a share as reconciled.

Open modeling decisions (refine here)

  • Itinerary : Trip — one itinerary holds many trips, or exactly one in Phase 0? Rec: 1 : many, but Phase 0 authors one trip per file.
  • Entry → Location optional FK (entries group under a leg) vs. entries only under Trip. Rec: optional FK.
  • Personal vs. shared entries — express "this is Effie's flight" via Location–Member (leg membership) + Finance–Member (who pays), or add a direct Entry–Member link? Rec: leg + finance links; add Entry–Member only if needed.
  • Allocation input — explicit shareAmount vs. sharePct vs. equal-split helper. Rec: store amounts; offer equal-split + percentage as helpers.
  • subType ("Activity Type") — free text vs. controlled enum per type. Rec: controlled list per type, with an "other" escape.
  • Overlap threshold — fixed at ≥2 days, or configurable? Rec: fixed at 2 for Phase 0.
