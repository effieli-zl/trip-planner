Trip Planner — Phase 0 Data Model

The itinerary is the single source of truth. One uniform entry record; booking status drives what the UI reveals — not the schema.

Standalone, file-backed (trips/<slug>/itinerary.json), validated against a versioned JSON Schema. Durable contract a future app/connector can consume.

schemaVersion 0.1.0 · draft for review

Structure & relationships

Composition (◆ embedded object) and reference (↝) relationships. Everything hangs off the trip; the entry embeds its detail objects.

Reading it: ◆ = composition (object lives inside its parent). ↝ = reference by id. An entry embeds all of its detail objects (times, place, ownership, price, type fields); the only cross-reference is a personal entry's ownership.contributorId pointing at a trip.contributors[] member.

Open modeling decisions (refine here)

  • bookingRef hoisted to entry level vs. per-type. Rec: hoisted — cross-cutting, main IDP target.
  • date as day anchor, day-index derived, vs. explicit integer day. Rec: date.
  • Times canonical in start/end; typeFields non-temporal — removes duplication. Rec: yes.
  • Multi-currency: store native, no auto-conversion in Phase 0. Rec: yes.
  • Attachments reference-only (no binaries committed). Rec: yes.
  • Compare-options via sibling entries + archived, no special structure. Rec: yes.
