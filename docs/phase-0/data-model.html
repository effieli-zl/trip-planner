Trip Planner — Phase 0 Data Model

A Trip is the wrapper — it owns the budget and the people. It contains one or more Itineraries (plan variants) you draft and compare during planning; each itinerary's cost rolls up and is measured against the trip budget. The selected itinerary is the source of truth for the actual trip.

Standalone, file-backed (trips/<slug>/trip.json), validated against a versioned schema. A durable contract a future app/connector can consume.

schemaVersion 0.3.0 · draft for review

Hierarchy (containment)

Parent → child ownership. Budget and Members sit on the Trip; plan detail lives inside each Itinerary variant.

Trip — wrapper · id, name, budget target, base currency, people
Itinerary 1 : many — a plan variant · status: draft · selected · archived; dates (rollup); cost compared to trip budget
Location — start/end date · overlap warning when 2 locations overlap ≥2 days (dismissible)
Entry — type: Lodging · Transportation · Activity · Dining; subtype, start/end date-time
Finance — total, estimate⇄confirmed, paid, unpaid (derived)
Member 1 : many — people on the trip · role: Owner · Coordinator · Participant · Follower; traveling flag

Entity-relationship diagram

Crow's-foot notation. | one · < many. Dashed = optional reference. Junctions (Location–Member, Finance–Member) resolve the many-to-many links.

Why the split: Trip holds the budget and the people; each Itinerary is a self-contained plan variant whose cost (Σ Finance) is compared against the trip budget. LocationMember and FinanceMember divide legs and costs across trip members without duplicating entries.

Relationship (junction) tables

JunctionLinksCard.Extra fieldsPurpose
Trip–MemberTrip · Member* : *role, travelingWho is on the trip & in what capacity. (Held on Member in Phase 0, since members are trip-scoped.)
Location–MemberLocation · Member* : *arrival?, departure?Which members are at which leg — supports converging trips (people arrive/leave on different days).
Finance–MemberFinance · Member* : *shareAmount / sharePct, isPayee, settledBill allocation — split a cost per person, mark who fronted it (payee), track settlement.
Bill allocation logic (Finance–Member): each Finance row can be divided among members by explicit amount or percentage (helper: equal split across traveling members). Members flagged isPayee paid it. "Who owes whom" = each member's shareAmount minus what they paid, netted across the trip; settled marks a share reconciled. Only the selected itinerary's allocations are treated as real.

Open modeling decisions (refine here)

  • One selected itinerary at a time (others draft/archived); allocations count only for it. Rec: yes.
  • Budget granularity — single trip target vs. per-category (lodging/transport/…). Rec: single target in Phase 0; per-category later.
  • Dates — rollup at itinerary level + optional trip target window. Rec: yes.
  • Members are trip-level and shared across all itineraries. Rec: yes.
  • Personal vs. shared entries — express via Location–Member + Finance–Member, or a direct Entry–Member link? Rec: leg + finance links; add Entry–Member only if needed.
  • subType — controlled enum per type vs. free text. Rec: controlled list per type, with "other".
